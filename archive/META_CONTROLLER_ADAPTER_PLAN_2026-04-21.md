# Meta-Controller Adapter Plan - 2026-04-21

## Purpose

This note selects the first old F-drive project to adapt after the V0.14 interface freeze.

The goal is not to merge another project into the experiment. The goal is to add one narrow, local adapter that strengthens falsification without changing the causal core:

> bounded workspace -> habit-safe gate -> planner-necessity gate -> gated read/write `ControlDecision`

## Decision

Start with a `sel-lab` benchmark registry adapter.

Do not start with runtime, memory, planner, or predictor integration.

Rationale:

- `sel-lab` helps the weakest near-term engineering need: benchmark organization, controller registry, ablation matrix discipline, and transfer reporting.
- It does not need to change `WorkspaceState`, `ControlDecision`, environment dynamics, habit/planner behavior, or memory semantics.
- A registry/reporting adapter is much easier to falsify than a module donor. If it changes experiment results, the adapter is doing too much.
- It keeps all old F-drive projects reference-only and avoids dependency on half-finished code.

## Candidate Ranking

| rank | candidate | adapter role | decision | reason |
|---:|---|---|---|---|
| 1 | `F:\sel-lab` | benchmark registry and ablation matrix | start now | lowest behavioral risk, directly improves falsification |
| 2 | `F:\cognitive-execution-engine` | event/replay log adapter | later | useful, but can expand runtime scope too early |
| 3 | `F:\diff_world` | surprise/delta predictor adapter | later | changes a causal input signal, so it needs stronger isolation |
| 4 | `F:\SDAS-LLM\sdas-structure-memory` | gated memory adapter | later | changes memory semantics and can hide read/write causal effects |
| 5 | `F:\system_stability` | drift/recovery metric adapter | later | useful but not blocking while current metrics work |
| 6 | `F:\SDAS` | structure/habit adaptation adapter | defer | powerful but prototype-heavy and likely to distort the V0 claim |

## First Adapter Scope

Name:

- `sel_lab_benchmark_adapter`

Location if implemented:

```text
F:\unified-sel\experiments\meta_controller\adapters\sel_lab_benchmark.py
F:\unified-sel\tests\test_meta_controller_adapter_sel_lab.py
```

Documentation target:

```text
F:\unified-sel\META_CONTROLLER_ADAPTER_SEL_LAB_RESULT_2026-04-21.md
```

Allowed responsibilities:

- define benchmark suites as local Python data structures
- group controllers into families
- select profiles, seeds, masking settings, and read-cost stress cases
- produce a deterministic run matrix for `run_experiment`
- format comparison tables from existing report metrics

Forbidden responsibilities:

- no cross-repo import from `F:\sel-lab`
- no `pip install -e F:\sel-lab`
- no change to controller decisions
- no change to environment transition rules
- no change to metric definitions
- no hidden filtering of failed runs
- no special treatment for `planner_necessity_loose_controller`

## V0.14 Contract Mapping

The first adapter does not produce `WorkspaceState` and does not consume `ControlDecision` directly.

It operates one level outside the causal core:

| V0.14 surface | adapter interaction |
|---|---|
| `WorkspaceState` | none; must remain generated by `env.py` / local monitors |
| `ControlDecision` | none; must remain generated by controllers |
| controller registry | reads local controller names from `baselines.py` or a local registry wrapper |
| benchmark profiles | selects existing profiles such as `v01` and `v02` |
| ablations | selects existing mask flags and baseline controllers |
| metrics | consumes existing report rows only |

This means the adapter should be observational and organizational. If implementation requires modifying `meta_controller.py`, it is probably the wrong first adapter.

## Initial Benchmark Suites

### Suite 1: Mainline Acceptance

Purpose:

- verify that B3 still beats fixed rule on the core pressure profile.

Runs:

- profile: `v01`
- seeds: `0, 1, 2`
- train episodes: `240`
- eval episodes: `60`
- controllers:
  - `fixed_rule_controller`
  - `planner_necessity_loose_controller`
  - `habit_safe_set_h2_controller`
  - `random_controller`

Acceptance:

- B3 success `>= 0.99`
- B3 drift no worse than fixed-rule guard band
- B3 planner calls below fixed rule
- B3 memory reads within fixed + `1.0`

### Suite 2: Transfer

Purpose:

- verify that the learned dominance law is not just a profile-specific script.

Runs:

- profile: `v02`
- seeds: `0, 1, 2`
- controllers:
  - `fixed_rule_controller`
  - `planner_necessity_loose_controller`
  - `habit_safe_set_h2_controller`

Acceptance:

- B3 success at least fixed-rule success
- B3 drift at most fixed-rule drift
- B3 planner calls below fixed rule

### Suite 3: Cost Stress

Purpose:

- verify that planner savings matter when computation cost increases.

Runs:

- profile: `v01`
- read cost: `0.20`
- seeds: `0, 1, 2`
- controllers:
  - `fixed_rule_controller`
  - `planner_necessity_loose_controller`

Acceptance:

- B3 total reward above fixed rule
- B3 success stays `>= 0.99`

### Suite 4: Signal Masking

Purpose:

- verify that the causal core depends on the expected control signals.

Runs:

- profile: `v01`
- seed: `0`
- controller: `planner_necessity_loose_controller`
- masks:
  - none
  - `surprise`
  - `memory`
  - `conflict`
  - `drift`
  - core signals together

Acceptance:

- masking conflict, memory, or surprise must degrade behavior
- masking core signals together must strongly degrade behavior
- drift masking may remain neutral in V0, but must be logged

## Data Shape

The adapter should expose a small local object model:

```text
BenchmarkSuite
  name
  profile
  seeds
  train_episodes
  eval_episodes
  read_cost
  controllers
  masks
  acceptance_rules
```

Each expanded run should be reducible to existing CLI parameters.

No adapter object should contain environment internals or controller-specific thresholds.

## Local Tests Required

Before any implementation is accepted:

1. suite expansion is deterministic
2. all controller names exist in the local baseline registry
3. all selected profiles exist
4. masks map to existing masking behavior
5. acceptance rules can evaluate a minimal synthetic result table
6. adapter can generate commands or run specs without importing `F:\sel-lab`
7. existing `test_meta_controller_protocol.py` still passes

## Integration Gate

The adapter may be implemented only if all are true:

- it writes only inside `F:\unified-sel`
- it does not import or install `F:\sel-lab`
- it does not modify the controller, environment, or metrics to make itself work
- it adds tests before broad benchmark execution
- it can be removed without affecting one-off `run_experiment` commands
- it produces the same metrics for an equivalent manually invoked run

## Rejection Conditions

Reject or rewrite the adapter if:

- it starts duplicating controller logic
- it creates a second source of truth for metrics
- it requires donor project cleanup before local tests can run
- it hides per-seed failures behind averaged tables
- it changes the positive B3 result by changing the experimental surface
