# Meta-Controller V0.14 Interface Contract - 2026-04-21

## Purpose

This document freezes the V0 causal core interface after the positive B3 results.

The interface exists to prevent premature integration of old F-drive prototypes. Future modules may be adapted into the experiment only if they implement this contract and preserve the causal tests.

Current mainline controller:

- `planner_necessity_loose_controller`

Core claim supported so far:

> A two-gate controller, combining habit-safe initiation with planner-necessity estimation, can learn a transferable dominance law over habit/planner/memory use.

## Causal Core

The V0 causal core has four layers:

1. bounded workspace state `g_t`
2. habit-safe gate
3. planner-necessity gate
4. read/write gates and macro-action output

The controller may choose habit only when:

- habit-safe gate passes
- planner-necessity gate is inactive

Otherwise it falls back to planner while preserving read/write decisions.

## Workspace State Contract

`WorkspaceState` is the only input to the meta-controller.

Required fields:

| field | type | meaning | source requirement |
|---|---|---|---|
| `task_progress` | float | normalized progress through current episode/task | monotonic or task-phase normalized |
| `surprise` | float | recent prediction error / unexpected outcome | computed from predictor or monitor |
| `uncertainty` | float | confidence deficit for near-term action success | bounded scalar |
| `control_cost_estimate` | float | expected computational/control cost pressure | includes planner/read/write burden |
| `memory_relevance` | float | whether memory may affect current action | must distinguish delayed query states |
| `conflict_score` | float | disagreement between habit and planner/candidate action | key dominance signal |
| `invariant_violation` | float | drift, identity, or long-horizon consistency violation | bounded scalar |
| `recent_failure_source` | float | recent action/control failure indicator | nonzero after observed failure |

Rules:

- Fields must be bounded floats.
- Missing fields must be explicitly masked, not silently defaulted by an adapter.
- New fields may be added only behind an ablation.
- `conflict_score`, `memory_relevance`, and `surprise` are currently causal-critical.
- `invariant_violation` is retained even though current V0 tests show it is redundant.

## Module Contract

### Habit Module

Required methods:

- propose an action from the current observation/context
- update from action success/failure

Behavioral role:

- cheap default policy
- may be wrong after regime shifts
- must not read long-term memory directly unless the meta-controller selects a read action

### Planner Module

Required methods:

- propose an action under deliberation
- optionally condition on explicit memory read value

Behavioral role:

- expensive recovery and disambiguation path
- may use forward simulation or oracle-like planning in synthetic tests
- planner cost must be measured

### Memory Module

Required methods:

- read selected long-term fact/case
- write selected fact/case
- expose read/write counts and useful read/write labels in benchmark settings

Behavioral role:

- not always-on context stuffing
- read/write must be gated by the meta-controller
- memory contribution must remain measurable through precision/recall

### Predictor / Monitor

Required methods:

- estimate expected success or next-state consistency
- emit surprise/prediction-error signal

Behavioral role:

- provide `surprise`
- may provide `uncertainty`
- must be local and cheap relative to planner

## Controller Output Contract

The meta-controller returns a `ControlDecision`:

| field | type | meaning |
|---|---|---|
| `name` | str | macro-action name |
| `dominant_module` | str | `habit` or `planner` |
| `memory_read` | bool | whether to read long-term memory before acting |
| `memory_write` | bool | whether to write current observation/result |
| `broadcast` | bool | whether a control change enters shared workspace |
| `deliberation_precision` | float | strength of deliberate control |

Current valid macro-actions:

- `habit`
- `planner`
- `habit_read`
- `planner_read`
- `planner_read_write`
- `write_then_habit`
- `write_then_planner`

Adapters may add new macro-actions only if they can be reduced to these fields for evaluation.

## Training Label Contract

### Habit-Safe Labels

Label:

- `habit_safe(g_t) -> bool`

Current V0 generator:

- clone environment at state `t`
- force habit for a short horizon
- positive if habit succeeds without drift increase

Required properties:

- generated by counterfactual or held-out validation, not by the controller's own choice alone
- horizon documented
- positive rate logged
- failure labels include memory misses, regime-shift failures, and drift increase

### Planner-Necessity Labels

Label:

- `planner_necessary(g_t) -> bool`

Current V0 generator:

- compare short-horizon rollout value for first-step planner vs first-step habit
- positive if planner value exceeds habit value by a margin

Required properties:

- value margin documented
- planner cost included
- memory read/write costs included
- label positive rate logged

## Required Metrics

Every integration candidate must report:

- `task_success`
- `total_reward`
- `planner_calls`
- `memory_reads`
- `memory_writes`
- `memory_read_precision`
- `memory_read_recall`
- `memory_read_false_positive_rate`
- `drift_under_horizon`
- `switch_latency`
- `recovery_after_surprise`
- `arbitration_regret`
- `safe_set_positive_rate`
- `safe_label_positive_rate`
- `necessity_positive_rate`
- `necessity_label_positive_rate`

Acceptance for V0-compatible controllers:

- success at least fixed-rule success on transfer or `>= 0.99` where fixed is perfect
- drift no worse than fixed-rule guard band
- memory reads within fixed-rule + 1.0 unless explicitly testing memory-cost tradeoffs
- planner calls below fixed-rule
- signal masking must degrade performance when core signals are removed

## Required Ablations

At minimum:

- fixed-rule replacement
- random controller
- habit-only
- planner-always
- memory-always
- mask surprise
- mask memory
- mask cost
- mask conflict
- mask drift
- mask core signals

Current causal-critical signals:

- `conflict_score`
- `memory_relevance`
- `surprise`

Current redundant signal:

- `invariant_violation` in V0/v01/v02

This redundancy is environment-specific and must not be removed from the interface.

## Integration Gate

A donor project can move from reference to adapter only when all are true:

- clean source boundary identified
- no editable install or sibling-path import
- adapter writes only inside `F:\unified-sel`
- adapter produces `WorkspaceState` or consumes `ControlDecision`
- local tests cover adapter behavior
- V0 acceptance table still passes
- signal masking still shows core-signal degradation
